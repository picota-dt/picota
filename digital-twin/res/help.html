Introduction

Picota is a platform for building and running Digital Twins — virtual models of real-world systems that evolve over time and respond to changes in the environment.

At the heart of Picota is its domain-specific language (DSL), which lets users define structured models that describe the components of a system, the variables they depend on, and the way these elements change over time. This modeling layer is clear, compact, and expressive.

But Picota is not just a DSL — it also includes a runtime engine that brings these models to life. Once a model is defined, Picota can use real-world data to build and operate the Digital Twin: ingesting time-series data, interpreting variables, and estimating or forecasting unknown values.

In Picota, it's important to distinguish three main concepts.

Descriptive Model

A Descriptive Model is written using the Picota DSL. It defines the structure of the system you want to simulate:

  • What exists: Reality and Subject
  • What is measured: Variable
  • What should be estimated: InferenceModel
  • How often it updates: Resolution

By itself, this model doesn't estimate or predict anything — it's just a formal description of the system.

Inference Model

An Inference Model is created when the Picota RT processes the Descriptive model and combines it with historical data. It builds a trained model that can:

  • Learn from past data
  • Estimate unknown or future values
  • Be used for simulations or online predictions

The inference model is what makes the Digital Twin operational.

Digital Twin

A Digital Twin in Picota is an operational entity that encapsulates one or more Inference Models. It is built and deployed using Picota DT, a web service that packages the trained models and turns them into a functional unit capable of simulating, predicting, and adapting to real-world data in real time.

A Digital Twin represents a live system, able to combine multiple trained models that cover different variables, resolutions, or components. It can coordinate their outputs, deliver integrated predictions, and run full-system simulations either continuously or on demand.

Summary

You write a Descriptive Model using the Picota DSL. Then you build and use an Inference Model using the Picota RT.

Tip: Think of the Descriptive Model as the blueprint, and the Inference Model as the engine that runs estimations and predictions.

This documentation will guide you through the core concepts of Picota, how to define your own models, and how to connect them to real data. You will find practical examples, reusable patterns, and ready-to-use templates to help you get started quickly.

How Picota Works

A Picota-based Digital Twin is developed in three stages, following a clear and modular lifecycle.

Descriptive Model Definition

The first step is to write a Descriptive Model using the Picota DSL. This model acts as a blueprint: it defines what the system looks like, what entities it contains, and which variables are of interest.

In this phase, the user declares:

  • Reality: the overall environment (e.g. a city)
  • Subject: the key entities within the system (e.g. a park
  • Variable: measurable values such as temperature, or resource usage
  • DigitalTwin: the artifact that is being created

This descriptive model contains no logic for prediction—it is purely declarative. It defines what the system is, not how it behaves.

Inference Model Training

Once the Descriptive Model is in place, the Picota RT uses it—along with historical data—to build an Inference Model. This model is trained to estimate or predict the value of variables marked for inference in the DSL.

This step involves:

  • Loading structured input data (usually from CSV files)
  • Training inference models based on the historical data
  • Creating an operational model that can estimate or forecast some variables

The Inference Model is the driving component that brings the Digital Twin to life—it knows how to generate predictions using both learned patterns and new inputs.

Digital Twin Deployment

Once one or more Inference Models have been trained, they are packaged into a Digital Twin — a web service that runs the models in a live environment. The Digital Twin can now operate using these Inference Models to produce outputs for selected variables in different modes:

  • Offline simulation: apply the Digital Twin to test data or hypothetical scenarios, replaying or exploring alternative outcomes
  • Online operation: feed the Digital Twin with continuously updated input data (e.g. hourly or daily) and generate real-time or periodic predictions
  • Model recalibration: update or fine-tune the underlying inference models as more historical data becomes available or as system behavior evolves

Online operation allows the Digital Twin to function continuously, reflecting the latest conditions of the real system and adapting over time — without requiring constant manual intervention.

Summary

  • Picota DSL is used to define the system — the Descriptive Model (structure, variables, update rules).
  • Picota RT builds and runs the Inference Model — trained from historical data to perform estimation and forecasting.
  • Picota DT is a web service that packages one or more inference models into an operational Digital Twin — ready for simulation, monitoring, and real-time prediction.
Tip: Think of the Picota DSL as the “what” (structure), Picota RT as the “how” (learning and inference), and Picota DT as the “where” it runs (deployment and operation).

Getting Started

This sections guides you through the creation of your first digital twin using the Picota DSL. You will model a city with a public park and estimate how many people are present in the park based on noise level and calendar features.

1. Open the Picota Editor

Go to quassar.io/picota. This is the official modeling interface for Picota. Click “New” to open a blank editor.

2. Edit the Descriptive Model

Paste the following Picota DSL code into the editor:

Reality MyCity
	Variable dayOfWeek is TimeFeature(DayOfWeek)
	Variable season is TimeFeature(QuarterOfYear)

	Subject CentralPark
		Variable noiseLevel is Numeric
		Variable occupancy is Numeric

DigitalTwin ParkTwin
	subject = MyCity.CentralPark

	Resolution
		value = 1
		scale = days

	Inference
		variable = occupancy
		Estimation

This model defines:

  • MyCity as the modeling context
  • dayOfWeek and season as automatic time features derived from the date
  • CentralPark as the subject with:
    • noiseLevel: a measurable variable from sensors
    • occupancy: the target variable to estimate
  • ParkTwin as the twin that will estimate occupancy daily

3. Commit the Model

Click “Commit” to validate the model and create a versioned release. Only committed models can be used to build digital twins.

4. Forge the Digital Twin

Click “Forge” and follow the assistant:

  • Upload your CSV file with date, noiseLevel, occupancy
  • Ensure dayOfWeek and season are extracted from the date column
  • Select occupancy as the variable to estimate
  • Run the training

5. Use Your Digital Twin

After training, your twin is ready to:

  • Estimate occupancy from new noise levels
  • Operate in online mode with daily inputs
  • Export predictions for monitoring or alerts

Summary

To create your first digital twin in Picota:

  1. Open the editor at quassar.io/picota.
  2. Define a Reality with TimeFeatures and a Subject for the park.
  3. Declare measurable variables like noiseLevel and occupancy.
  4. Configure a DigitalTwin to estimate occupancy daily.
  5. Commit the model and use Forge to upload training data and run the estimation.
  6. Start simulating or operating the twin online with updated data.
Tip: Start simple. You can always extend the model later with additional variables, subjects, or forecasting logic.

DSL Reference

This section provides a comprehensive reference for all constructs in the Picota DSL. It includes syntax, usage, and detailed explanations of core features such as layered variables and prototype subjects.

Overview of Constructs

  • Variables
  • Subjects
  • Reality
  • DigitalTwin

Variables

A Variable defines a data point associated with either a Reality or a Subject. Variables represent measurable, derived, or categorical values. All variables must explicitly declare their type.

Syntax

Variable ::= "Variable" IDENTIFIER "is" Type [ "Layered" ]
					[ Dependency ]
					{ Layer }

Dependency ::= INSIDE "depends" "=" IDENTIFIER { IDENTIFIER }

Layer ::= INSIDE "Layer" "(" "values" "=" LITERAL ")" [ "is" "Disjoint" ] { Layer }

Type ::= "Numeric" | "Enumerated" "(" LITERAL { "," LITERAL } ")"
                   | "TimeFeature" "(" TimeFeatureKind ")"

TimeFeatureKind ::= "DayOfWeek" | "DayOfMonth" | "MonthOfYear" | "WeekOfYear"
                   | "QuarterOfYear" | "Year" | "IsWeekend" | "IsHoliday"

IDENTIFIER ::= ? valid identifier name ?
LITERAL ::= '"' { character } '"'
INSIDE ::= ? newline and indentation ?

Notes

  • TimeFeature variables are only allowed in the Reality scope.
  • Layered can only be used with Numeric variables.
  • depends must reference a variable declared in the same Subject or in the containing Reality.

Examples

Numeric Variable

Represents quantitative values such as temperature, usage, occupancy, or count. It is the most commonly used type.

Variable temperature is Numeric

Enumerated Variable

Represents a fixed list of possible string values. Use this type when the variable has a limited set of categories.

Variable status is Enumerated("open", "closed", "maintenance")

Time Feature

Represents values automatically derived from the input date column. These variables are used to capture temporal structure (e.g. weekday/weekend patterns) and are only allowed in the Reality scope.

Available values include: DayOfWeek, DayOfMonth, MonthOfYear, WeekOfYear, QuarterOfYear, Year, IsWeekend, IsHoliday

Variable dayOfWeek is TimeFeature(DayOfWeek)

Layered Variable

A basic layered variable uses one layer. For example, to model consumption by region:

Variable consumption is Numeric Layered
Layer(value = "region.txt")

If region.txt contains:north, south, east west

Expected inputs:

consumption.north, consumption.south, etc.

Nested Layers

You can nest layers to represent hierarchical structures, such as regions and product types:

Variable consumption is Numeric Layered
Layer(value = "region.txt")
  Layer(value = "product.txt")

Expected inputs:

consumption.north.food, consumption.south.energy

Multiple Independent Layers

You can also define multiple independent branches of layers:

Variable consumption is Numeric Layered
Layer(value = "region.txt")
  Layer(value = "product.txt")
Layer(value = "category.txt")

Expected inputs:

  • consumption.north.food, consumption.south.energy
  • consumption.residential, consumption.industrial

Disjoint Layered Variable

Use is Disjoint when layer values represent alternative measurements, not aggregable parts of a whole. For example:

Variable temperature is Numeric Layered
  Layer(value = "types.txt") is Disjoint

If types.txt contains: min, max, average

Expected inputs:

temperature.min, temperature.max, temperature.average

Variable Dependencies

In the Picota DSL, a variable can declare its dependency on one or more other variables using the depends keyword. This is a semantic hint used during model interpretation and training to specify that the value of one variable is functionally or statistically related to others.

Declaring dependencies does not alter the structure of the model or change the order of evaluation, but it informs the inference engine about which variables are relevant for estimation or prediction. This helps guide the learning process and improve model explainability.

Example:

Variable waterUse is Numeric
  depends = occupancy temperature dayOfWeek

This declaration expresses that waterUse is expected to vary based on occupancy, temperature, and the day of the week — allowing the model to incorporate behavioral and environmental factors during estimation.

Rules and Notes

  • Every Variable must declare its type: Numeric, Enumerated, or TimeFeature.

    TimeFeature variables may only be declared inside a Reality, not within a Subject.

    Variable names must be unique within the same Reality or Subject block.

    Tip: Use TimeFeature variables like dayOfWeek or monthOfYear to add calendar-awareness to your models without requiring manual date parsing.

  • Only Numeric variables can be layered.

    Enumerated and TimeFeature types do not support layering.

    All expected paths (e.g. consumption.north.food) must be present in the dataset if used for inference or prediction.

  • Enumerated values must be declared inline as a list of strings.

    These values are static and cannot be layered or loaded from external files.

  • You can use depends to indicate input variables for estimation.

    Any referenced variable must be visible — either within the same Subject or in the parent Reality.

    TimeFeature and Layered variables can be used in dependencies.

    depends is optional. If omitted, the system infers dependencies from the structure of the training data.

    Tip: Declare depends when you want to guide model logic explicitly or enforce known causal relationships.

Subjects

A Subject defines a measurable entity inside a Reality. It is the fundamental unit of simulation in Picota: all variables to be estimated or predicted belong to a subject.

There are two types of subjects:

  • Regular subjects: define a single named instance. Use these when the entity is unique in the model, such as a specific park, building, or zone.
  • Prototype subjects: define a reusable structure for many entities of the same type. Instances are created automatically from an external list of identifiers (e.g., a CSV file). Use these when modeling repeated structures like rooms, sensors, or machines.

Both types of subject can declare one or more variables. These variables are local to the subject unless they reference a global variable from the enclosing Reality using depends.

Syntax

Subject          ::= INSIDE "Subject" IDENTIFIER
                     { Variable }

PrototypeSubject ::= INSIDE "Subject" IDENTIFIER "is" "Prototype"
                     INSIDE "identifiers" "=" LITERAL
                     { Variable }

Examples

Regular Subject

This defines a single named subject with its own variables. Regular subjects are used when the entity is unique in the model (e.g., one specific park).

Subject CentralPark
  Variable noiseLevel is Numeric
  Variable occupancy is Numeric

Prototype Subject

Prototype subjects allow you to model many instances of the same structure, using identifiers from an external file.

Subject TrafficSensor is Prototype
  identifiers = "sensors.csv"
  Variable vehicleCount is Numeric
  Variable averageSpeed is Numeric

If sensors.csv contains:

id
S001
S002
S003
S004

The system will create these subject instances at runtime:

  • TrafficSensor[S001]
  • TrafficSensor[S002]
  • TrafficSensor[S003]
  • TrafficSensor[S004]

Rules and Notes

  • Each subject, whether regular or prototype, must have a unique name within the Reality.
  • Prototype subjects allow the model to scale efficiently by avoiding repetition.
  • The identifiers file should contain one unique ID per instance (e.g. sensor_001, sensor_002).
Use a regular subject when:
  • The entity is unique in the model (e.g., a single park or building)
  • You want to configure the digital twin with specific resolution or logic
  • You need to combine with prototype-based entities in the same reality
Tip: Regular subjects are ideal when you want full control over the variables and simulation of a single entity.
Use a prototype subject when:
  • You have many entities of the same type and structure
  • You want to load identifiers from a file (e.g., sensors, rooms, stations)
  • You want to apply inference to each instance individually
Tip: Use prototype subjects to keep your models scalable and DRY (Don't Repeat Yourself). The system handles instantiation and identity resolution automatically.

Reality

Reality is the top-level construct in Picota. It defines the context in which all Subjects exist. While subjects are the entities that we simulate, the reality acts as the environment — providing global variables and shared temporal or structural information.

A Reality contains:

  • Global variables, such as environmental indicators, time-derived features, or macroeconomic factors
  • Subjects, defined directly or via prototypes to instantiate from a dataset

Examples

SmartCity

Includes time features and socioeconomic variables.

Reality SmartCity
  Variable dayOfWeek is TimeFeature(DayOfWeek)
  Variable ipc is Numeric

  Subject PublicPark
    Variable occupancy is Numeric

FactoryFloor

Includes environmental and scheduling variables.

Reality FactoryFloor
  Variable shift is TimeFeature(DayOfWeek)
  Variable temperature is Numeric
  Variable electricityPrice is Numeric

  Subject Machine is Prototype
    identifiers = "machines.csv"
    Variable errorRate is Numeric

GreenhouseNetwork

Includes seasonal and climate-related variables.

Reality GreenhouseNetwork
  Variable season is TimeFeature(MonthOfYear)
  Variable rainfall is Numeric
  Variable humidity is Numeric

  Subject Greenhouse is Prototype
    identifiers = "greenhouses.csv"
    Variable yield is Numeric
    Variable temperature is Numeric

Rules and Notes

The reality block is intended to define global variables and contextual features that apply across all subjects. These variables are not simulated on their own — they serve as shared inputs for estimation and prediction within subjects.

Use the reality to declare:

  • Time-related variables like day of week, season, or holiday flags
  • Environmental measurements such as temperature, rainfall, or humidity
  • Macroeconomic indicators such as inflation index, fuel price, or policy flags
Tip: Use a mix of global variables to represent the structure of your system — time features for periodicity, numeric variables for measurements, and custom indicators for policy, cost, or external drivers.
Tip: Declare time features and environmental data at the reality level so that all subjects can reference them without duplication.

DigitalTwin

A DigitalTwin represents the simulation configuration for a specific subject. It defines how subjects behaves over time, at what resolution it should be updated, and which variables are to be estimated or predicted. A digital twin references a subject and binds it to an inference model and resolution of execution.

Syntax

DigitalTwin ::= "DigitalTwin" IDENTIFIER
                       INSIDE "subject" "=" IDENTIFIER "." IDENTIFIER
                       Resolution
                       { InferenceModel }

Resolution ::= INSIDE "Resolution"
                       INSIDE "value" "=" NUMBER
                       INSIDE "scale" "=" TIMEUNIT

InferenceModel ::= INSIDE "InferenceModel"
                       INSIDE "variable" "=" IDENTIFIER
                       ( Estimation | Prediction )

Estimation ::= INSIDE "Estimation"
Prediction ::= INSIDE "Prediction"
					   INSIDE "timeHorizon" "=" NUMBER

Examples

Basic Estimation

This twin estimates water usage daily based on inputs like occupancy or weather.

DigitalTwin ParkTwin
  subject = SmartCity.PublicPark

  Resolution
    value = 1
    scale = days

  InferenceModel
    variable = waterUse
    Estimation

Prediction with Time Horizon

This twin forecasts occupancy three days ahead using a prediction model.

DigitalTwin ForecastTwin
  subject = MyHotel.MainBuilding

  Resolution
    value = 1
    scale = days

  InferenceModel
    variable = occupancy
    Prediction
      timeHorizon = 3

Digital Twin for Prototyped Subjects

This digital twin is linked to a prototype subject, an inference model will be created for each sensor instance.

Subject TrafficSensor is Prototype
  identifiers = "sensors.csv"
  Variable vehicleCount is Numeric
  Variable averageSpeed is Numeric

DigitalTwin SensorTwin
  subject = SmartCity.TrafficSensor

  Resolution
    value = 1
    scale = days

  InferenceModel
    variable = averageSpeed
    Estimation

Rules and Notes

  • Each digital twin must reference a subject using reality.subject notation.
  • The resolution block defines how often the twin updates its internal state (e.g., once per day).
  • Each inference model must point to a declared variable within the subject.
  • Use Estimation for predicting current unknown values.
  • Use Prediction for forecasting future values based on a time horizon.
  • A single digital twin may include multiple inference models, each for a different variable.
Tip: Use multiple digital twins when you want to simulate the same subject at different resolutions or with different inference goals.
Tip: Keep inference models focused: each should estimate or predict a single variable, based on relevant inputs.
